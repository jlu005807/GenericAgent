# SOP 格式规范 v1.0

**本文档定义 GenericAgent 标准操作流程 (SOP) 文件的统一格式，以支持技能的提取、共享与安装。**

---

## 1. 概述

SOP（Standard Operating Procedure）是 GenericAgent 存储技能的核心格式。本规范确保：
- 所有 SOP 文件具有统一的结构，便于解析器自动提取
- 技能可在不同 GA 实例间无缝共享
- 元数据完整，支持版本控制和检索

---

## 2. 文件命名规范

```
<skill_name>_sop.md       # 技能类 SOP（如 browser_automation_sop.md）
<utility_name>.py          # 工具类脚本（如 ocr_utils.py）
```

- 使用小写字母和下划线
- 技能类 SOP 必须以 `_sop.md` 结尾
- 工具类脚本使用 `.py` 后缀

---

## 3. 必需结构（Mandatory Sections）

每个 SOP 文件 **必须** 包含以下部分，按顺序排列：

### 3.1 YAML Front Matter（元数据头）

```yaml
---
name: <技能名称>
version: <版本号>  # 语义化版本 x.y.z
description: <一句话描述技能功能>
author: <作者或来源>
created: <YYYY-MM-DD>
updated: <YYYY-MM-DD>
tags: [tag1, tag2, tag3]  # 用于检索和分类
trigger: <触发条件描述>  # 何时激活此 SOP
input: <输入描述>
output: <输出描述>
---
```

### 3.2 标题与描述

```markdown
# <技能名称> SOP

**触发**：<触发条件> | **禁用**：<不适用场景>
**核心原则**：<一句话概括执行准则>
```

### 3.3 前置条件（可选但推荐）

```markdown
## 前置条件
- 环境要求、依赖工具、权限需求等
```

### 3.4 工作流程（核心）

```markdown
## 工作流程

### Step 1: <步骤名称>
- 具体操作指令
- 代码示例（如适用）

### Step 2: <步骤名称>
...
```

**要求**：
- 每个步骤必须是**具体可执行的动作**
- 包含代码示例时使用代码块
- 步骤间有清晰的依赖关系说明

### 3.5 错误处理（推荐）

```markdown
## 常见错误与处理

| 错误场景 | 处理方式 |
|----------|----------|
| ... | ... |
```

### 3.6 权限边界（如涉及）

```markdown
## 权限边界
- 无需批准：<操作列表>
- 需要批准：<操作列表>
- 绝对禁止：<操作列表>
```

### 3.7 状态机/流程图（复杂技能推荐）

```markdown
## 状态机

​```
开始 → 步骤1
  → 成功 → 步骤2
  → 失败 → 错误处理 → 重试/终止
​```
```

---

## 4. 可选结构（Optional Sections）

以下部分根据技能复杂度选用：

| 章节 | 用途 | 何时使用 |
|------|------|----------|
| `## 配置参数` | 列出可调参数及其默认值 | 有可配置项时 |
| `## 示例` | 完整使用示例 | 复杂技能 |
| `## 依赖` | 依赖的其他 SOP 或工具 | 有依赖时 |
| `## 版本历史` | 变更记录 | 需要追溯时 |
| `## 参考` | 相关文档链接 | 有外部参考时 |

---

## 5. 格式规则

### 5.1 Markdown 规范
- 使用标准 CommonMark 语法
- 代码块指定语言类型（`python`、`bash`、`javascript` 等）
- 表格对齐，便于阅读

### 5.2 语言
- 中英文均可，但**同一 SOP 内保持一致**
- 推荐：中文描述 + 英文代码注释

### 5.3 长度控制
- 理想长度：50-200 行
- 超过 300 行考虑拆分为子 SOP

---

## 6. 解析器兼容性要求

为支持自动解析，SOP 文件需满足：

1. **YAML Front Matter 必须存在**且格式正确
2. **标题层级清晰**：`#` → `##` → `###`，不跳级
3. **触发条件**必须在 YAML 或正文中明确标注
4. **代码块**必须可由正则匹配提取
